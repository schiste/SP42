# SP42 Scoring Policy Layout

This document turns the scoring constitution into a concrete repository layout.

## Directory Layout

Scoring policy is organized by lifecycle:

```text
configs/scoring/
  active/
  candidate/
  suggested/

evals/scoring/
  profiles/
  fixtures/
```

## Policy Lifecycles

### `active/`

Contains the currently accepted policy files used by the scoring system.

Rules:

- must declare `lifecycle: active`
- must declare `domain`
- must declare `evaluation_profile`
- must satisfy the scoring policy schema

### `candidate/`

Contains policies under review.

Rules:

- must declare `lifecycle: candidate`
- may be generated by maintainers or tools
- must pass evaluation before becoming active

### `suggested/`

Contains machine-generated or exploratory policy suggestions.

Rules:

- must declare `lifecycle: suggested`
- may be incomplete for activation
- may never be promoted silently

## Evaluation Profiles

Evaluation profiles live in:

```text
evals/scoring/profiles/
```

Each profile defines:

- the domain
- required gates
- fixture-set paths
- fairness tracking expectations
- performance budget

## Fixture Sets

Recommended fixture layout:

```text
evals/scoring/fixtures/<domain>/<wiki_id>/
  regression.yaml
  ranking.yaml
  invariants.yaml
  fairness.yaml
```

These files are intended to support the evaluation gates required by the scoring constitution.

## Naming Guidance

Policy files should use names that describe both wiki and domain.

Examples:

- `frwiki-vandalism.yaml`
- `wikidata-structured-data.yaml`
- `enwiki-article-quality.yaml`

Candidate and suggested files should include a short intent suffix where useful.

Examples:

- `frwiki-vandalism-tighten-identity-cap.yaml`
- `frwiki-vandalism-duplicate-burst-experiment.yaml`

## Source of Truth

The source of truth for active operational scoring behavior is the human-readable policy file in `configs/scoring/active/`.

Rust code defines the scoring engine mechanics. Policy defines active operational tuning.
