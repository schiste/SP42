# Repository Cleanup Plan

This plan keeps the repository easier for external contributors to understand,
build, and review without changing product scope.

## Goals

- Keep `main` buildable and releasable.
- Make common local checks fast and obvious.
- Separate contributor workflows from maintainer-only deployment/release work.
- Reduce stale docs, duplicate build paths, and confusing generated artifacts.

## Phase 1: Contribution Baseline

- Keep `README.md`, `CONTRIBUTING.md`, `SECURITY.md`, `GOVERNANCE.md`, and
  `CODE_OF_CONDUCT.md` current.
- Require PR descriptions to include validation notes.
- Keep `.github/CODEOWNERS` aligned with protected areas.
- Use `good first issue` only for issues that can be solved without deployment
  credentials or architectural context.

## Phase 2: Build And CI Cleanup

- Make `./scripts/check-focused.sh` the recommended pre-PR check.
- Keep `./scripts/ci-all.sh` as the maintainer/full-CI path.
- Avoid adding new shell branching when `xtask` can own cross-platform logic.
- Pin CI tool versions when release artifacts depend on them.
- Keep clean builds explicit with `--clean`; do not purge `target/` by default.

## Phase 3: Documentation Cleanup

- Move architecture decisions into ADRs instead of README sections.
- Keep deployment docs in focused files:
  - `docs/DESKTOP_DISTRIBUTION.md`
  - VPS package README generated by `./scripts/package-vps.sh`
- Keep README focused on quick start, project status, and navigation.
- Remove or update docs that describe obsolete commands.

## Phase 4: Generated Artifact Hygiene

- Keep generated build output out of git.
- Keep Tauri sidecar binaries ignored in `crates/sp42-desktop/src-tauri/binaries/`.
- Keep local runtime state in ignored runtime directories.
- Audit untracked files before every release commit.

## Phase 5: Contributor-Friendly Issues

- Label issues by area and difficulty.
- Split large roadmap items into reviewable tasks.
- Add reproduction steps and expected validation commands to bug issues.
- Reserve maintainer-only labels for release, deployment, and security work.

## Phase 6: Release Readiness

- Require green CI before tagging.
- Build unsigned desktop artifacts first.
- Add signing/notarization only after unsigned artifacts are stable.
- Keep Wikimedia Cloud VPS deployment runbooks explicit and separate from
  local development instructions.
